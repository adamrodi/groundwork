# Groundwork

## What this is

Salim runs a small landscaping business. He pays $40/month for Jobber but only uses it for invoicing — the rest of the platform (scheduling, CRM, quoting) is overkill for a one-person operation. Groundwork replaces Jobber with a lightweight PWA that does exactly what he needs: track clients, log jobs, and send invoices.

Salim's real-world workflow is: quote a job → do the work → invoice the client → send it via text or email → get paid.

Salim is the only user for now and gives ongoing feedback. Long-term this becomes a SaaS product for other landscaping/trades businesses.

## Non-goals (MVP)

- **No multi-user / teams** — single user (Salim). No employee accounts, roles, or permissions.
- **No quoting** — Salim quotes verbally or via text. Formal quote workflow comes post-MVP.
- **No scheduling / calendar** — Salim manages his own schedule outside the app.
- **No tap-to-pay in the PWA** — NFC requires native iOS SDK. For on-site payments, Salim uses the Square POS app directly. Tap-to-pay moves into Groundwork when the native iOS app is built.

**Tailwind v4 note:** Configured via `@tailwindcss/vite` in `vite.config.ts`. There is no `tailwind.config.js` — that's intentional for v4. Theme lives in `src/index.css` with three distinct blocks:
- `@theme { ... }` (top) — project color overrides (green primary, etc.). Edit this for custom theme changes.
- `@theme inline { ... }` and `:root`/`.dark` blocks — generated by shadcn/ui. Edit via `shadcn` CLI or carefully by hand.

Animation utilities come from `tw-animate-css` (imported in `src/index.css`).

## Architecture constraints

- **No custom backend for MVP.** Everything goes through Supabase SDK + Edge Functions. No Express/FastAPI server until post-MVP.
- **Client-side first.** Prefer client-side logic (PDF generation, form validation, data fetching via Supabase JS SDK) over server-side. Use Edge Functions only when there's no alternative (webhooks, sending email).
- **Supabase RLS is the auth boundary.** Row Level Security policies on every table enforce data access — the client talks directly to Supabase, not through a middleware layer.

## Database schema

All tables are in Supabase (Postgres). RLS is enabled on all of them. All `id` columns are `uuid` (default `gen_random_uuid()`). All timestamps are `timestamptz`.

- `clients` — id, user_id (uuid, FK → auth.users), name (text), email (text), phone (text), address (text), created_at
- `jobs` — id, user_id (FK → auth.users), client_id (FK → clients), description (text), date (date), status (`pending`|`complete`), created_at
- `line_items` — id, job_id (FK → jobs), description (text), quantity (numeric), unit_price (numeric, cents)
- `invoices` — id, user_id (FK → auth.users), job_id (FK → jobs), client_id (FK → clients), invoice_number (serial, Postgres-generated), status (`draft`|`sent`|`paid`), total (numeric, cents), sent_at, paid_at, created_at

Money is stored in **cents** (integer) to avoid floating-point issues. Divide by 100 for display. The `payment_method` column does not exist yet — will be needed when payments are built.

## Conventions

- **Naming** — PascalCase for file names and component names. camelCase for variables and functions.
- **Page components** — one per file in `src/pages/`, named after the route (e.g. `Clients.tsx` for `/clients`). Each page uses `export default function ComponentName()`.
- **Class merging** — use `cn()` from `src/lib/utils.ts` (wraps `clsx` + `tailwind-merge`) for conditional Tailwind classes
- **Icons** — use `lucide-react`
- **shadcn/ui** — components in `src/components/ui/` are owned code; edit them directly when needed. Add new components with `npx shadcn@latest add <component>`. Config is in `components.json`. Components use `radix-ui` (unified package) as their primitive layer — do not install individual `@radix-ui/*` scoped packages.
- **Tailwind** — the developer is learning Tailwind for the first time; prefer clear, readable class usage over clever one-liners
- **Path alias** — `@/` maps to `src/` (configured in `tsconfig.app.json`, `tsconfig.json`, and `vite.config.ts`)
- **Data fetching** — no library chosen yet; decide when the first real page is built

## Environment & config

- `.env.local` holds Supabase credentials — never commit it (covered by `*.local` in `.gitignore`)
- `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` — required for the Supabase client
- `VITE_APP_URL` — base URL for public invoice links (e.g. `https://groundwork.vercel.app`); also set in Vercel environment variables
- Supabase Edge Functions will live in `supabase/functions/` — each function is a Deno TypeScript module; secrets set via Supabase dashboard or `supabase secrets set`
